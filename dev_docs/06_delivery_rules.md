# 交付规则

## 1. 目标

本文档用于防止开发过程偏离既定方案，明确什么属于重大变更、什么时候必须先更新文档，以及交付前如何核对实现。

## 2. 变更分级

### 2.1 重大变更

以下任一情况都属于重大变更：

1. 修改技术栈主组件
2. 修改模块边界或进程模型
3. 新增、删除或改变 Tauri 命令语义
4. 新增、删除或改变 Sidecar 接口语义
5. 新增、删除或改变 SQLite 表结构、关键字段、索引、状态枚举
6. 修改搜索范围、搜索字段或搜索结果行为
7. 修改主界面信息架构
8. 修改人工确认触发条件或动作集合
9. 修改路径规则、冲突规则、RAG 写入门槛
10. 修改设置页分组或将其升级为插件系统、多人协作后台

### 2.2 小改动

以下情况属于小改动：

1. 不改变契约含义的文案调整
2. 不改变主流程的样式微调
3. 不改变外部行为的内部重构
4. 不改变字段语义的代码组织调整

## 3. 重大变更流程

发生重大变更时必须按以下顺序执行：

1. 识别受影响文档
2. 先更新 `dev_docs`
3. 重新确认变更内容
4. 再进入实现

禁止流程：

1. 先改代码后补文档
2. 仅口头说明，不更新 `dev_docs`
3. 在实现中引入未文档化的重大行为

## 4. 小改动限制

小改动可以直接实现，但必须同时满足：

1. 不改变已有接口语义
2. 不改变数据结构
3. 不绕过已定业务规则
4. 不改变 UI 主流程

## 5. 开发前检查清单

开始实现前必须逐项确认：

1. 当前任务是否影响 `01-06` 任一文档契约
2. 是否新增字段、状态、接口、目录或交互路径
3. 是否改变 AI 输入输出或路径规则
4. 是否改变搜索行为或设置项
5. 如果任一项为“是”，先更新文档

## 6. 验收前检查清单

实现完成后必须逐项核对：

1. 是否符合 `01_product_contract.md`
2. 表结构是否符合 `03_data_model.md`
3. 接口是否符合 `04_api_contracts.md`
4. UI 是否符合 `05_ui_spec.md`
5. 是否存在未文档化的重大行为差异

## 7. 文档更新规则

1. 文档更新必须保持中文编写。
2. `development_plan.md` 继续保留，不删除。
3. `00-06` 是正式规范源，新增重大能力时必须补充对应文档。
4. 如果多个文档同时需要更新，必须在同一提交中完成，避免中间态冲突。

## 8. 交付门槛

以下情况不得进入实现或不得提交为完成：

1. 存在与产品契约冲突的设计
2. 表结构与接口文档不一致
3. UI 方案与 `05_ui_spec.md` 不一致
4. 引入了新的重大行为但没有更新文档
5. 搜索路径展示、打开文件、打开所在位置、复制路径四项能力任一缺失

## 9. 文档审计要求

每轮重大迭代至少完成一次文档审计，确认：

1. 文档之间没有职责重叠和互相冲突
2. 产品规则、架构边界、接口、数据模型、UI 规范可以一一对应
3. 所有“必须”行为都能追溯到唯一文档源
