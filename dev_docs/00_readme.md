# AI 文件整理工具文档入口

## 1. 目的

本目录用于维护 AI 文件整理工具的正式开发文档。`development_plan.md` 保留为阶段性总纲，`00-06` 文档负责实现约束与交付治理。

这些文档的目标是让后续开发始终对齐最初规定，避免在架构、接口、数据模型、UI 主流程和业务规则上发生无文档漂移。

## 2. 阅读顺序

建议按以下顺序阅读：

1. `development_plan.md`
   回答“产品按什么阶段推进”
2. `01_product_contract.md`
   回答“什么必须做，什么不能做”
3. `02_system_architecture.md`
   回答“系统怎么拆，模块如何协作”
4. `03_data_model.md`
   回答“本地存储和向量记忆如何落地”
5. `04_api_contracts.md`
   回答“模块间如何通信”
6. `05_ui_spec.md`
   回答“界面长什么样，用户能做什么”
7. `06_delivery_rules.md`
   回答“什么改动必须先更新文档”

## 3. 文档职责矩阵

| 文档 | 负责内容 | 不负责内容 |
| --- | --- | --- |
| `development_plan.md` | 阶段目标、里程碑、总体方向 | 具体接口、具体表结构、具体 UI 细节 |
| `01_product_contract.md` | 产品目标、非目标、刚性约束 | 表结构和接口字段定义 |
| `02_system_architecture.md` | 技术栈、模块边界、进程模型、数据流 | 详细 SQL DDL |
| `03_data_model.md` | SQLite 与 Chroma 数据模型、状态、索引 | UI 布局和视觉规范 |
| `04_api_contracts.md` | Tauri 命令、Sidecar 契约、请求响应结构 | 页面布局细节 |
| `05_ui_spec.md` | 信息架构、核心页面、交互约束 | 存储实现细节 |
| `06_delivery_rules.md` | 变更分级、文档更新规则、交付检查 | 产品功能新增决策 |

## 4. 文档优先级

当文档之间出现范围重叠时，按“更具体的文档优先于更一般的文档”解释：

1. 产品边界与不可偏离规则，以 `01_product_contract.md` 为准。
2. 数据结构、字段、索引、状态枚举，以 `03_data_model.md` 为准。
3. 接口字段、错误码、超时语义，以 `04_api_contracts.md` 为准。
4. 界面布局、搜索行为、交互动作，以 `05_ui_spec.md` 为准。
5. 模块边界、进程模型、调用链，以 `02_system_architecture.md` 为准。
6. 变更分级、开发前检查、验收前检查，以 `06_delivery_rules.md` 为准。
7. `development_plan.md` 只负责路线和阶段目标，不作为字段、接口、表结构或 UI 的最终来源。

## 5. 术语表

1. `受管文件`
   已经被本工具导入、整理、记录并可被搜索的文件或文件夹。
2. `批次`
   一次拖拽导入产生的一组处理任务。
3. `任务`
   针对一个文件或文件夹的独立处理单元。
4. `人工确认`
   因低置信度、冲突、未知类型或规则校验失败而触发的用户干预。
5. `浅采样`
   对文件夹只读取名称、部分子项名与类型分布，不做全量深解析。
6. `RAG 样本`
   可参与相似检索的高质量整理记录。
7. `重大变更`
   影响架构、接口、数据模型、UI 主流程或业务规则的改动。

## 6. 使用规则

1. 新增功能前，先判断是否影响 `01-06` 中任一文档。
2. 若影响范围达到“重大变更”标准，必须先更新文档再开发。
3. 小改动可以直接实现，但不得改动已定义契约。
4. 实现验收必须回查 `01-06` 对应条目，不允许以代码为唯一真相源。
